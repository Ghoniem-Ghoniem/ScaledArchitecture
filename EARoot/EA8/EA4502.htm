As Solution Architect,<br /></p><p><br /></p><p>I want to enable logging infrastructure for our application to trace any unusual incident or error present in the project and to understand what's going on with the application.<br /></p><p><br /></p><p>Enabling logging engine can be designed in two different approaches:<br /></p><p><ol><li>Each micro-service can handle it's logs, which will apply the logs segregation, in this case, the logs will be scattered in different DB Schemas and it's very hard to collect the details in one view.    </li><br /></p><p><li>A dedicated Logging micro-service that can handle all the logs for all other micro-services, in that case we can easily build a front-end application over this micro-service to read, monitor, and visualize the actions and errors that occurred in this system      </li><br /></p><p></ol><br /></p><p>From above, we will go with the second approach as it's clearly will fulfill our requirement <br /></p><p><br /></p><p><br /></p><p>Consider the following in the logging model:<br /></p><p><br /></p><p>Separate data model (DB/Schema)<br /></p><p>Configurable Logging level: <br /></p><p>Business/Technical/Exceptions<br /></p><p>Begin of operation/Stepwise/End of operation to enable traceability<br /></p><p>Reports and analytics for the logging<br /></p><p>Operational model for the model <br /></p>
